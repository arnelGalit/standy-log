# Daily Standup Tracker Project

## Project Overview
A single-page React application for teams to log daily standup entries with localStorage persistence.

## Tech Stack
- React 18+ with TypeScript
- Vite for build tooling
- CSS Modules for styling
- localStorage for persistence
- No external UI libraries (built from scratch)

## Core Types

```typescript
// Main entry type
interface StandupEntry {
  id: string;
  name: string;
  date: string;          // ISO date string (YYYY-MM-DD)
  yesterday: string;
  today: string;
  blockers: string;
}

// For creating new entries (id generated by storage)
type NewStandupEntry = Omit<StandupEntry, 'id'>;

// Storage error handling
type StorageErrorType = 'QUOTA_EXCEEDED' | 'PARSE_ERROR' | 'UNKNOWN';

interface StorageError {
  type: StorageErrorType;
  message: string;
}

interface StorageResult<T> {
  success: boolean;
  data?: T;
  error?: StorageError;
}
```

## Storage Utility Functions

```typescript
// Get all entries from localStorage with validation
getEntries(): StorageResult<StandupEntry[]>

// Save new entry with auto-generated UUID
saveEntry(entry: NewStandupEntry): StorageResult<StandupEntry>

// Delete entry by ID
deleteEntry(id: string): StorageResult<boolean>

// Get entries from last N days, sorted by date desc
getRecentEntries(days?: number): StorageResult<StandupEntry[]>
```

## Key Features Implemented
- Form to capture standup data with validation
- Display last 7 days of entries grouped by date
- "Today" and "Yesterday" labels for recent dates
- localStorage persistence with error handling
- Entry deletion with confirmation
- Error boundary for crash protection
- Loading states during localStorage operations
- Dismissible error banner for storage errors
- Form retains name/date after submission for convenience
- Responsive two-column layout (form + list)
- Empty state messaging

## Design Decisions

1. **Result Pattern for Storage**: All storage functions return `StorageResult<T>` rather than throwing errors, allowing graceful handling of localStorage issues.

2. **Entry Validation**: Storage validates entry shape when loading to handle corrupted data gracefully.

3. **Date Grouping**: Entries are grouped by date in the list view with human-readable headers (Today, Yesterday, or full date).

4. **Form UX**: Name and date fields persist after submission since users often submit multiple entries for the same person/date.

5. **ErrorBoundary**: Uses class component (required for error boundaries) with retry functionality.

6. **Sorting**: Entries sorted by date descending, then by name ascending for same-date entries.

7. **CSS Modules**: Each component has its own scoped CSS module to prevent style conflicts.

## Development Guidelines
- Use functional components with TypeScript (except ErrorBoundary)
- Implement proper error boundaries
- Add loading states for localStorage operations
- Use proper TypeScript types (no `any`)
- Handle edge cases gracefully
- Keep components small and focused
